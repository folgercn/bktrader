# 运行时恢复 / 接管扩展编码规范

## 目的

本文档用于把本轮“运行时恢复稳定性改造”的经验，正式固化为后续编码规范。

适用范围：
- research / live 运行时逻辑
- 恢复链路
- 历史仓位接管
- 被动平仓路径
- 交易所同步与对账
- AI / Codex 参与的后续修改

目标不是重复业务逻辑，而是约束以后所有相关改动必须遵守统一的扩展规则，避免系统再次退化回“能跑策略，但不具备安全恢复能力”的状态。

---

## 总原则

以后凡是涉及以下内容的修改：
- 恢复
- 接管
- 平仓
- dispatch
- 交易所同步
- WebSocket 重连
- state / session / runtime 相关字段

都必须先回答这三个问题：

1. **谁是真实事实源？**
2. **哪些只是缓存态？**
3. **当前改动会不会让未验证的恢复态直接参与交易？**

如果这三个问题回答不清楚，不允许直接编码。

---

## 规则 1：事实源层级必须明确

以后任何相关改动，必须显式区分：

### 1.1 事实源
以下内容属于事实源：
- 交易所 REST 对账结果
- 已确认的本地持仓记录（且未被交易所否定）
- 已确认的订单 / 成交结果

### 1.2 缓存态
以下内容默认视为缓存态，不允许直接当成事实源：
- session.State
- livePositionState
- recoveredPosition
- virtualPosition
- WebSocket 推送的临时状态

### 1.3 推导态
以下内容属于推导态：
- planIndex
- strategy decision
- intent / proposal
- runtime health summary

### 强制要求
- 不允许让缓存态静默覆盖事实源
- 不允许让推导态反向写回事实源并当成真实仓位使用
- 如果缓存态与事实源冲突，必须进入显式状态（如 stale / conflict / error）

---

## 规则 2：恢复状态必须显式，不允许继续堆 flag 组合

以后扩展恢复逻辑时，必须优先扩展显式状态机，而不是继续新增零散布尔字段来“猜测系统当前状态”。

### 必须遵守
恢复 / 接管相关改动必须明确：
- 当前状态名称
- 状态允许的动作
- 状态禁止的动作
- 状态转移条件

### 最低要求
每个恢复态至少要回答：
- 允许开仓吗？
- 允许平仓吗？
- 允许补保护单吗？
- 允许 auto-dispatch 吗？
- 必须人工 review 吗？

### 禁止行为
- 不允许通过“多看几个 flag”来隐式推断状态
- 不允许 unresolved recovery 静默落回正常 RUNNING 语义

---

## 规则 3：恢复态在完成对账前，不得参与自动交易

以后任何涉及恢复、接管、历史仓位继续运行的改动，都必须遵守：

> 恢复出来的仓位，只有在执行上下文完整、并且已经完成与交易所事实的对账之后，才允许参与自动动作。

### 必须遵守
- 未完成对账，不允许 auto-dispatch
- 未完成对账，不允许被动平仓自动提交
- 未完成对账，不允许继续推进正常策略计划

### 允许的安全行为
- 标记为 stale / conflict / error
- 进入 close-only takeover
- 等待人工处理
- 只做观察和记录，不做执行

---

## 规则 4：WS 和 REST 的职责不能混淆

以后所有对账、恢复、重连相关改动，必须明确：

### 4.1 WS 的职责
- 提供时效性更新
- 反映运行中的实时变化
- 作为事件流触发器

### 4.2 REST 的职责
- 提供权威校验
- 作为启动、接管、重连后的事实确认入口
- 作为恢复链能否继续动作的门禁依据

### 强制要求
- 不允许把 WS 状态直接当作恢复态的最终事实依据
- WebSocket 重连后，必须触发 REST 级别的对账或验证
- 不允许因为“WS 又连上了”就静默恢复健康状态

---

## 规则 5：最终执行边界必须保持严格

以后任何新增订单类型、平仓逻辑、接管态执行路径，都必须在“最终提交边界”重新校验一次语义完整性。

### 必须校验的内容
- 订单分类是什么（normal entry / normal exit / recovered passive close / 其他）
- 是否必须具备 reduce-only 语义
- HEDGE / ONE_WAY 模式下的 payload 是否正确
- positionSide 是否符合目标仓位语义
- 当前状态是否允许执行

### 强制要求
- 不允许只在 proposal / intent 阶段校验，然后在最终 submit 前放松约束
- 最终提交边界必须仍然有独立的安全护栏

---

## 规则 6：禁止“顺手修”跨越多个问题域

以后无论是人工开发还是 AI / Codex 开发，都必须按 issue / PR 控边界。

### 必须遵守
- 一个 PR 只解决一个清晰问题域
- 不允许在修 recovery metadata 时顺手改状态机
- 不允许在修 reconcile 时顺手重构 execution strategy
- 不允许“顺手统一字段”导致历史语义改变

### 推荐方式
- 一 issue 一分支
- 一 issue 一 PR
- 开发可以并行，review / merge 必须串行

---

## 规则 7：测试必须先覆盖运行时现实，再扩展逻辑

以后凡是涉及恢复/接管/被动平仓/对账的改动，必须优先考虑测试，而不是先改逻辑再看看会不会炸。

### 最低要求
必须至少覆盖：
- DB-backed takeover
- exchange-only takeover
- DB vs exchange mismatch
- existing open exit order 不重复提交
- partial fill + restart
- passive-close payload 语义
- unresolved recovery 阻断自动动作

### 禁止行为
- 不允许只补 helper 级测试，而不覆盖 service-level 行为
- 不允许只测 happy path
- 不允许在没有 recovery regression test 的情况下扩展恢复语义

---

## 规则 8：AI / Codex 参与开发时必须遵守的附加要求

以后凡是让 AI / Codex 修改恢复、接管、平仓、同步逻辑，都必须附带如下约束：

### 必须明确给 AI 的信息
1. 本次只改哪个 issue
2. 主入口文件和函数在哪里
3. 哪些行为是允许的
4. 哪些行为绝对不能发生
5. 必须补哪些测试

### 必须要求 AI 输出
- root cause
- 修改点
- 行为变化
- 新增/修改测试

### 严禁
- AI 在一个 PR 里顺手处理相邻 issue
- AI 自作主张扩大 scope
- AI 为了“让测试通过”而放宽安全校验
- AI 在未解释事实源层级的前提下修改恢复逻辑

---

## 规则 9：编码前必须先过扩展检查表

以后任何相关改动，编码前都必须先自检下面问题：

### 扩展检查表
- 当前改动涉及的事实源是什么？
- 当前改动涉及的缓存态是什么？
- 是否新增了恢复态？如果有，动作矩阵是否明确？
- 是否会影响 auto-dispatch？
- 是否需要对账门禁？
- 是否会影响最终 submit payload？
- 是否已经补了 recovery regression test？
- 如果服务在中途重启，这段逻辑还能成立吗？

只要其中一项回答不清楚，就不应该直接提交代码。

---

## 推荐的后续执行方式

以后凡是继续扩展这一块，建议固定使用下面流程：

1. 先写清楚问题属于哪个问题域
2. 先标出事实源、缓存态、推导态
3. 先补测试或至少先定义测试场景
4. 再改代码
5. 再验证恢复 / 接管 / 平仓链没有回归
6. 最后 review 时，必须从“运行时恢复安全”角度再看一次，而不只看代码是否能跑

---

## 最终要求

以后所有与运行时恢复安全相关的修改，都必须服从本规范。

如果某次改动违反了本文档中的规则，即使代码能运行、测试也暂时通过，仍然应视为设计层面不合格，需要回退或重做。

本文档应与以下文档一起使用：
- `docs/research-runtime-recovery-review-plan.md`
- `docs/runtime-recovery-stabilization-summary.md`

这三份文档共同构成后续 runtime recovery / takeover / passive close 扩展的基线规范。

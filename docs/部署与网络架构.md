# BKTrader 部署与网络架构拓扑图

本文档专门用于记录 bktrader 项目从代码合并到上线的流量回路、部署方式，以及各节点间的端口映射说明。由于长期不看容易遗忘，在这里进行归档总结。

## 一、 自动化部署（CI/CD）流程

整个部署过程主要由 GitHub Actions 的 `cd.yml` 接管，采用分离式架构运作：

1. **后端构建与发布 (GitHub 云端完成)**：
   - 当代码合入 `main` 分支时触发。
   - 读取 `Dockerfile` 打包最新的 Go 后端代码，生成包含 AMD64 和 ARM64 跨平台架构的 Docker 镜像。
   - 镜像被直接推送到 GitHub 的私有镜像库 GHCR 中。

2. **后端更新上线 (Macmini 自托管服务器完成)**：
   - GitHub Action 控制你的 Macmini 开始工作。
   - Macmini 使用本地的 `deploy.sh` 脚本和 `docker-compose.prod.yml`，把网上的最新镜像拉回本地。
   - 借助 Docker Compose 启动拆分后的后台容器：`platform-api` 对外监听 **`8080`**，`live-runner` 负责实盘恢复 / 对账 / session runtime，`notification-worker` 负责 Telegram 告警投递。
   - `platform-api` 使用 `BKTRADER_ROLE=api`，不启动 live 后台 worker；`live-runner` 使用 `BKTRADER_ROLE=live-runner`，不暴露 HTTP 端口。
   - CD 会按 diff 计算 `DEPLOY_SERVICES`，后端改动默认只重启对应服务；`live-runner` 相关改动会自动重启交易运行时。

3. **前端资源发布 (Macmini → IDC远端机)**：
   - 同上步骤，Macmini 本地从源码里进行 `npm run build` 编译项目。
   - 前端无需硬编码任何服务端域名的端口（`VITE_API_BASE=""`），采用**相对路径请求**的模式进行打包。
   - Macmini 将构建好的前端 `dist/` 里的资源，通过 `rsync` 借助 SSH 推送至远端公网服务器 `1.95.71.247` 的 `/var/www/bktrader` 目录下。

## 二、 核心网络通路与端口映射 (非常重要❗)

我们的系统前端运行在公网节点（1.95.71.247），而后端实际跑在私网节点（Macmini）。他们之间由 Nginx 与内网穿透（FRP隧道）相连，通路流向参考如下：

### 1. 流量大动脉 (API 请求) 
当用户打开前端页面，需要请求后台登录数据 `/api/login` 时：

`用户的浏览器` ➡️ `公网 Nginx服务器 (1.95.71.247)` ➡️ `内网穿透隧道 (FRP)` ➡️ `Macmini 本机后端`

### 2. 逐层网络揭秘：

- **第一棒：用户的浏览器**
  - **访问入口**：`https://trade.sunnywifi.cn:3088`
  - **发起请求**：浏览器根据页面，自动组合相对路径，向外发出请求 `https://trade.sunnywifi.cn:3088/api/login`。

- **第二棒：远端前哨台 (1.95.71.247 - Nginx)**
  - Nginx 监听着 `3088` 端口并挂载了 SSL 证书。
  - **规则 A (找页面)**：如果是发向根目录 `/` 的请求，Nginx直接去本地服务器的 `/var/www/bktrader` 目录取 `index.html` 丢给用户。
  - **规则 B (找数据)**：如果是前缀带有 `/api/` 的接口请求，Nginx 不做处理，直接使用 `proxy_pass` 黑盒丢给它所在这台机器上的 `127.0.0.1:3081` 端口。

- **第三棒：搬运工 (FRP 隧道)**
  - `3081` 端口在公网机器 `1.95.71.247` 上是被 FRP 占用的。
  - 它通过加密隧道，悄悄穿越了防火墙，把 `3081` 接收到的任何字节原封不动地搬运到了你的内网 **Macmini 本机的 `8080` 端口**。

- **第四棒：终点站 (Macmini - 后端)**
  - 你的 Macmini 使用 `docker-compose.prod.yml` 跑拆分后的后台服务。公网请求只进入 `platform-api:8080`；实盘运行、对账和告警由同一 compose 栈里的 worker 容器在后台处理。API 解析收到的数据后返回 `HTTPS 200`，结果沿着旧路返回给用户浏览器。

### 小结
- **开发前端时**：API 基础地址配 `""`，保持相对路径最香，切忌写死 `127.0.0.1:8080`，不然公网用户访问网页时浏览器会去请求他们自己的电脑。
- **排查网络不通时**：由于架构采用分离态，优先查 Macmini 服务是否活着，其次查隧道 `3081` 是否正常通透，最后查 Nginx `3088` 配置。

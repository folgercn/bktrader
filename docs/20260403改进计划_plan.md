# bkTrader 项目 Review 与 README 中文化

## 项目总览

bkTrader 是一个 **BTCUSDT 量化交易平台**，包含两部分：
1. **历史遗留**：Python 策略研究和回测脚本（`backTest.py`、`btcusdt_1min.py` 等）
2. **新平台**：Go 后端 + React 前端的实盘/模拟盘/回测一体化平台

当前处于 **Phase 1 MVP** 阶段，已实现策略注册、Paper 交易执行、订单/成交/持仓管理、净值快照、TradingView 图表数据接口脚手架。

## 🔍 设计改进建议

以下是我 review 整个代码库后发现的设计问题和改进建议，**按优先级分级**：

---

### 🔴 高优先级（架构/可靠性问题）

#### 1. `platform.go` 上帝文件问题（934 行）

[platform.go](file:///Users/fujun/node/bktrader/internal/service/platform.go) 承载了**所有业务逻辑**：策略管理、订单执行、Paper 交易引擎、PnL 计算、CSV 回放、K 线生成、图表标注等。这违反了单一职责原则，随着功能增长会越来越难维护。

> [!IMPORTANT]
> **建议**：拆分为多个 service 文件，例如：
> - `service/strategy.go` — 策略管理
> - `service/order.go` — 订单执行和成交处理
> - `service/paper.go` — Paper 交易引擎和回放逻辑
> - `service/pnl.go` — PnL 计算和账户快照
> - `service/chart.go` — K 线数据生成和图表标注
>
> `Platform` struct 可以保留作为门面（Facade），但内部委托给专门的 service。

#### 2. Repository 接口的类型不一致

[store.go](file:///Users/fujun/node/bktrader/internal/store/store.go) 中 `ListStrategies` 和 `CreateStrategy` 返回的是 `map[string]any`，而其他所有方法都返回强类型 domain struct。

```go
// 当前：不一致
ListStrategies() ([]map[string]any, error)
CreateStrategy(name, description string, parameters map[string]any) (map[string]any, error)

// 其他方法都是强类型
ListAccounts() ([]domain.Account, error)
```

> [!WARNING]
> **建议**：将 Strategy 方法也改为返回 `domain.Strategy` 或 `domain.StrategyVersion`，消除动态类型，便于编译期检查和 IDE 补全。

#### 3. 缺少 CORS 中间件

前端运行在 Vite devserver（通常是 `localhost:5173`），后端在 `localhost:8080`。当前 HTTP handler 中没有任何 CORS 处理：

```go
// router.go — 没有 CORS headers
func NewRouter(cfg config.Config, platform *service.Platform) http.Handler {
    mux := http.NewServeMux()
    // ...
    return mux  // 没有中间件包裹
}
```

> [!CAUTION]
> **建议**：增加 CORS 中间件，至少在开发环境下允许 `localhost:5173` 跨域访问。否则前端完全无法调通后端 API。

#### 4. 缺少错误类型和上下文传递

- 所有 handler 用字符串传递错误，没有结构化的错误码
- handler 和 service 之间没有 `context.Context` 传递（只有 paper session 内部用了 context）
- 无法区分「资源不存在」和「内部错误」，统一返回 500

> [!IMPORTANT]
> **建议**：
> - 定义 domain 错误（`ErrNotFound`, `ErrConflict` 等）
> - handler 根据错误类型返回合适的 HTTP 状态码（404、409 等）
> - 所有方法接受 `context.Context` 参数

### 🟡 中优先级（可维护性/工程实践）

#### 5. 缺少测试

整个项目没有任何测试文件。关键的 PnL 计算逻辑（`applyPnLFill`）、Paper 引擎（`placePaperSessionOrder`）、仓位管理（`applyPaperFill`）等都是高风险代码。

> **建议**：至少为以下核心函数编写单元测试：
> - `applyPnLFill` — 多空开仓、平仓、反转的 PnL 计算
> - `applyPaperFill` — 仓位增减逻辑
> - `translateReplayEvent` — CSV 回放事件转订单
> - `buildAccountSummary` — 账户汇总计算

#### 6. K 线数据是伪造的

[CandleSeries](file:///Users/fujun/node/bktrader/internal/service/platform.go#L313-L359) 方法生成的是假数据（固定公式的波动），不是真实市场数据：

```go
base := 68000.0
wave := float64((index%17)-8) * 18
drift := float64(index) * 2.5
open := base + drift + wave
```

> **建议**：标记为 TODO/STUB，并在 system-design.md 中明确数据源对接计划（Binance API、本地数据库等）。

#### 7. 标注数据是硬编码的

[ListAnnotations](file:///Users/fujun/node/bktrader/internal/service/platform.go#L280-L311) 返回两条硬编码的固定标注，应该改为从回测结果或订单数据动态生成。

#### 8. 缺少日志和可观测性

- 没有结构化日志（只有 `log.Printf` 启动日志）
- 没有请求日志中间件
- 没有 Prometheus metrics 端点
- 出错时无法追踪问题

> **建议**：引入 `slog`（Go 1.21+）或 `zerolog`，添加请求日志中间件，至少记录每个请求的方法、路径、状态码、耗时。

#### 9. 配置管理缺少验证

[config.go](file:///Users/fujun/node/bktrader/internal/config/config.go) 加载配置后没有任何验证。当 `STORE_BACKEND=postgres` 但 `POSTGRES_DSN` 为空时，会在运行时才报错。

> **建议**：`Load()` 后增加 `Validate()` 方法，启动前快速失败。

#### 10. 前端缺少路由和组件拆分

[main.tsx](file:///Users/fujun/node/bktrader/web/console/src/main.tsx) 是一个 536 行的单文件 React 应用，所有组件和逻辑都在同一个文件中。没有 React Router，没有状态管理，没有 API 客户端分层。

> **建议**：
> - 拆分组件到独立文件
> - 引入 React Router 支持多页面
> - 抽取 API 调用到 `api/` 层

### 🟢 低优先级（代码质量细节）

#### 11. `sync.Once` 的 ledger 缓存无法刷新

`loadReplayLedger()` 使用 `sync.Once`，CSV 文件只读取一次。如果文件更新，需要重启服务。

> **建议**：考虑基于文件修改时间的缓存失效机制。

#### 12. Paper session runner 的 ticker 间隔硬编码

```go
ticker := time.NewTicker(15 * time.Second)
```

> **建议**：改为可配置参数。

#### 13. 缺少优雅关闭（Graceful Shutdown）

`main.go` 直接调用 `server.ListenAndServe()`，没有信号处理和优雅关闭：

```go
// 当前 — ctrl+C 直接杀进程
if err := server.ListenAndServe(); err != nil {
    log.Fatal(err)
}
```

> **建议**：添加 `signal.NotifyContext` + `server.Shutdown(ctx)` 实现优雅退出。

#### 14. 遗留 Python 脚本未有组织

根目录散落着 5 个 Python 脚本（`backTest.py`、`btcusdt_1min.py` 等），建议统一移到 `research/` 或 `scripts/` 子目录。

# 前端日志查看台实施方案 (Task Breakdown for LLMs)

> **给接手此任务的控制台前端 LLM Agent 的提示：** 
> 按照此方案逐步执行。不要修改后端 API 服务代码！

此计划是基于现存 BKTrader React 前端架构（Zustand + TailwindCSS）给出的完整任务拆解。在改动过程中，务必随时关注 `Frontend-Design-System-Skill` 以及 `React-Polanyi-Interaction-Skill` 的严格规范。

---

## 阶段 1：开发流程与步骤拆解

### 第 1 步: 顶部系统状态日志持久化
**改动范围**: `web/console/src/store/useUIStore.ts`
- **你需要做的事**: 
  - 在这个文件上方实现类似 `readStoredConsoleNav` 的本地读取方法: `readStoredSystemLogs` 和 `writeStoredSystemLogs`，LocalStorage key 使用 `"bktrader-system-logs"`。
  - 修改 `useUIStore` 的初始化参数，使 `systemLogs: readStoredSystemLogs()`。
  - 在 `setError` 方法内（现在包含追加日志到 `systemLogs` 前排的逻辑），在 `return` 新款状态之前，先触发 `writeStoredSystemLogs(nextLogs.slice(0, 40))` 方法将数据留存在本地浏览器。
  - 在 `clearSystemLogs` 方法执行时，清空掉 localStorage 中的数组。
- **边界提示**: 不要去动 `SystemStatusMenu.tsx` 页面代码，它目前仅消耗状态即可，把改动收敛在 `useUIStore.ts` 这个单一可信源中。

### 第 2 步: 全局框架导航与占位
**改动范围**: `web/console/src/store/useUIStore.ts`, `web/console/src/layouts/WorkbenchLayout.tsx`, `web/console/src/components/layout/MainContent.tsx`
- **你需要做的事**: 
  - 修改全局状态导航树 `SidebarTab`，将原来的 `"monitor" | "strategy" | "account"` 追加值为 `"monitor" | "strategy" | "account" | "log"`。且相应调整 `readStoredConsoleNav` 的安全转换判定。
  - 前往 `WorkbenchLayout.tsx`，找到 `aside` 左侧边栏代码。使用图标 `FileText` 或 `Terminal`（来自 `lucide-react`），新增名为 `"日志查看台"` 的 `<SidebarItem>`。
  - 创建新文件 `web/console/src/pages/LogStage.tsx`。写一个简单的 `export function LogStage()` 占位框。
     - **UI 基准**: 外层使用 `<div className="h-full relative overflow-hidden bg-zinc-950/20 p-4">`。
  - 修改 `MainContent.tsx`，在现有的三个 tab 条件渲染之后追加判断 `{sidebarTab === 'log' && (<LogStage />)}`。

### 第 3 步: 搭建 LogStage 页面和数据汇聚层
**改动范围**: `web/console/src/pages/LogStage.tsx`
- **你需要做的事**: 
  - **组件内部状态设计**:
    1. `logType`: 筛选项 (all | alert | notification | timeline)
    2. `levelFilter`: 等级过滤 (all | critical | warning | info)
    3. `autoRefresh`: bool (默认 true)。
  - **Zustand 数据提取**:
    通过 `useTradingStore` 取出当前的 `alerts`, `notifications`, `liveSessions` 和 `signalRuntimeSessions`。
  - **数据汇聚聚合 (Mapper)**:
     这是核心！你需要编写一个 `useMemo` 计算器，把这些杂散的结构转换成一个统一的前端日志数据结构（注意由于没有后端统一的 API，我们暂时在前端进行）：
     ```typescript
     type ConsoleLogEvent = {
        id: string;
        source: "alert" | "notification" | "timeline";
        level: "critical" | "warning" | "info" | "debug";
        title: string;
        message: string;
        eventTime: number; // 使用转算出的 timestamp 时间戳来执行全局倒序
        metadata?: any;
     }
     ```
     - Alerts：直接映射，`eventTime` 取自 `Date.parse(alert.eventTime)`。
     - Notifications：通过内部包含的报警映射。
     - Timelines：由于时间线附属于特定 Session。你需要把 `accountStage`, `liveSessions` 内部结构中自带的 `.state.timeline` 的数组扁平化抽出来，封装成 `ConsoleLogEvent` 放到这一层。
  - **渲染截流防重绘**:
     如果是 `autoRefresh=false` 停止刷新模式，界面应该锁定现有的队列，不受背后 store 更新影响。

### 第 4 步: 日志列表的视觉构建
**改动范围**: `web/console/src/pages/LogStage.tsx` 及其附属的小型 UI 组件
- **你需要做的事**: 
  - 画出左侧/顶部的过滤控制栏，样式使用毛玻璃 `<div className="backdrop-blur-md bg-zinc-900/60 border border-white/5 rounded-xl ...">`。
  - 日志条目列表渲染 `logEvents.map()`。
  - 条目的色彩系统应用（极其关键）：警示文字不得用死板的 CSS 默认红，必须使用 Tailwind 的 `text-rose-400`；信息级使用 `text-zinc-400`；正常/就绪级使用 `text-emerald-400` 等等；时间用 `text-zinc-500 font-mono`。
  - **安全保护**: 由于可能会多达千条 timeline，使用简单的 `.slice(0, 500)` 保障 DOM 节点不超过上限。

## 阶段 2：留给后端的作业清单 (Backend Boundaries)
我们在第一阶段完全用前端的 Store 和局部 State 实现了类似系统，以此解渴。接下来的阶段 **不可交由前台 React 工程师或系统负责**，留给后端同事的改造点如下：

1. **增设标准化统一事件流 `/api/v1/logs/events`**：
   - 现有的将大量 Timeline 塞入 LiveSession state 中，极大地浪费了带状流量，降低了 HTTP 数据吞吐。
   - 需要提供按参数 `?limit=100&cursor=XXX&accountId=YY` 获取聚合后独立表格日志的接口。
2. **增设纯文本系统日志流 `/api/v1/logs/system`**：
   - 目前 golang 后端的 `slog` 只朝向 stdout 输出并且服务被 docker compose 管控。
   - 如果控制台需要看到这些纯系统 log，后端需要内置类似于 tail 或 file reader 的方案通过接口输送到前端，而前端 LogStage 将增加一个 "system" 的 source filter。
3. **增加 SSE / WebSocket 实时总线 (`/api/v1/logs/stream`)**:
   - `LogStage` 最适合使用 SSE 推送变化进行刷新，这能使得前端控制台能关掉冗余的 5s 定时 Polling 系统架构。

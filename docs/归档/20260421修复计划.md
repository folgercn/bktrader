# SyncLiveAccount / SyncLiveOrder 历史订单污染修复方案

## 问题总述

恢复/对账链路 (`ReconcileLiveAccount` → `reconcileLiveAccountExchangeOrder` → `applyLiveSyncResult`) 把交易所历史已平仓订单重新纳入当前 live runtime，并且 sync 缺少限速、幂等和状态边界，导致本地订单、fill、position 状态被污染。

---

## 一、已确认的代码级根因

### 根因 1：`FetchRecentOrders` 使用 `/fapi/v1/allOrders` 无条件拉取全量历史

> [!CAUTION]
> 这是核心污染源头。

**代码位置**: [live_registry.go:258-283](file:///Users/fujun/node/bktrader/internal/service/live_registry.go#L258-L283)

```go
func (a binanceFuturesLiveAdapter) FetchRecentOrders(...) ([]map[string]any, error) {
    // ...
    payload, err := doBinanceSignedRequest(http.MethodGet, resolved, "/fapi/v1/allOrders", params)
```

**问题判断**:
- Binance `/fapi/v1/allOrders` 返回包括所有终态订单（FILLED、CANCELLED 等）在内的 **全量历史**
- 虽然传了 `startTime`/`endTime`（lookback 24h），但 24h 内有大量历史已平仓订单
- 这些已平仓订单与 **当前 live session 无关**，却全部被 `reconcileLiveAccountExchangeOrder` 无差别处理

**对比**: `SyncLiveAccount` (`syncLiveAccountFromBinance`) 正确地只拉取 `/fapi/v1/openOrders`（当前 open orders）和 `/fapi/v2/positionRisk`（当前持仓风险），**不会** 把历史关闭的订单重新导入。问题出在 `ReconcileLiveAccount` 的 reconcile 路径。

---

### 根因 2：`reconcileLiveAccountExchangeOrder` 对非本系统订单无条件创建 recovered order

**代码位置**: [live.go:615-645](file:///Users/fujun/node/bktrader/internal/service/live.go#L615-L645)

```go
func (p *Platform) reconcileLiveAccountExchangeOrder(...) (domain.Order, bool, error) {
    order, found := index.match(exchangeOrderID, clientOrderID)
    if !found {
        order, err = p.createRecoveredLiveOrderFromExchange(account, binding, payload)  // ← 无条件创建
        // ...
    }
    // 然后无条件走 applyLiveSyncResult → finalizeExecutedOrder → applyExecutionFill
}
```

**问题判断**:
- 任何在 index 中匹配不到的 exchangeOrderID 都会被创建为 recovered order
- `createRecoveredLiveOrderFromExchange` 创建的订单缺少关键元数据：`decision_event_id`、`strategy_version_id`、`live_session_id`、`signal_kind` 全部为空
- 历史已 FILLED 订单通过 `buildLiveReconcileSyncResult` 生成 fills → `applyLiveSyncResult` → `finalizeExecutedOrder` → `applyExecutionFill` 修改 position
- 同一个历史订单可能在不同 reconcile 周期中被反复创建（虽然 DB 层有 fill dedup 机制，但 **order 层没有 dedup**）

---

### 根因 3：`collectLiveAccountReconcileSymbols` 收集范围过宽

**代码位置**: [live.go:490-544](file:///Users/fujun/node/bktrader/internal/service/live.go#L490-L544)

**问题判断**:
- 该函数从**所有本地订单**（包括已完成的）、**所有持仓**、**account snapshot positions/openOrders**、**所有 live sessions** 收集 symbol
- 导致 reconcile 按 symbol 遍历时，每个 symbol 都会调 `FetchRecentOrders` + `FetchRecentTrades` 各一次
- 10 个 symbol × 2 个 API = 20 次请求/reconcile，多次 reconcile 触发极易命中 Binance 429

---

- 无全局 rate limiter、无 per-account semaphore

### 根因 5：Fill dedup 存在但 Order 创建无 dedup

**代码位置**:

- Fill 层：`CreateFill` 有 `ON CONFLICT (order_id, exchange_trade_id)` 和 `ON CONFLICT (order_id, dedup_fallback_fingerprint)` — [store.go:611-691](file:///Users/fujun/node/bktrader/internal/store/postgres/store.go#L611-L691) ✅
- Fill 应用前：`filterExistingExecutionFills` 做内存去重 — [order.go:1024-1061](file:///Users/fujun/node/bktrader/internal/service/order.go#L1024-L1061) ✅
- Fill 数量限制：`limitExecutionFillsToRemainingQuantity` 限制不超量 — [order.go:1078-1097](file:///Users/fujun/node/bktrader/internal/service/order.go#L1078-L1097) ✅
- **Order 层：`CreateOrder` 直接 INSERT，无 `ON CONFLICT`，无 `exchangeOrderId` unique index** ❌

**问题判断**:
- Fill dedup 机制在 **同一个 order ID** 内是有效的
- 但 reconcile 每次对同一个 `exchangeOrderId` 创建 **新的 order ID** (`order-{timestamp}`)
- 因此 fill 的 dedup key 变了（`orderID|trade|tradeID`，orderID 不同了），fill 被重新创建
- position 被重复 apply

## 二、修复方案（六大修复域）

---

### 修复域 1：ReconcileLiveAccount 拉取范围限制

#### 1.1 `FetchRecentOrders` 结果过滤

**修复方法**:
在 `reconcileLiveAccountExchangeOrder` 入口添加 **前置过滤器**，只处理以下订单之一：
1. 本地已知的 `exchangeOrderId`（index 中存在）
2. 属于当前 open position 的相关 symbol **且** 订单状态为 open/partially filled
3. 被本地 live session 管理的 `clientOrderId`（即本系统提交的订单 ID）

**边界与约束**:
```
规则 R1.1: reconcile 禁止对 index 中不存在 且 订单终态(FILLED/CANCELLED/REJECTED) 
           的交易所订单创建 recovered order，除非满足以下全部条件：
           a. 当前 REST 持仓 quantity > 0 且 symbol 匹配
           b. 订单 side 与当前持仓方向一致（即可能是该持仓的开仓订单）
           c. clientOrderId 匹配本系统 order ID 格式

规则 R1.2: 历史已平仓订单（终态 + 对应 symbol 无当前持仓）永远跳过，
           不创建、不 enrich、不 apply
```

#### 1.2 `collectLiveAccountReconcileSymbols` 收紧

**修复方法**:
将收集逻辑限定为：
1. 当前 REST snapshot 中 `positions` 有持仓量的 symbol（来自 `liveSyncSnapshot.positions`）
2. 当前 REST snapshot 中 `openOrders` 的 symbol
3. 本地 `positions` 表 `quantity > 0` 的 symbol
4. RUNNING 状态的 live sessions 的 symbol

**删除**: 从本地 `orders` 表收集 symbol 的逻辑（即时间窗口内的历史 order symbol）

**边界与约束**:
```
规则 R1.3: reconcile symbol 候选集 = 
           { REST open position symbols } ∪ 
           { REST open order symbols } ∪ 
           { local position symbols with qty > 0 } ∪ 
           { RUNNING live session symbols }
           禁止从历史已完成订单推断 symbol。
```

#### 1.3 Lookback 时间窗口收紧

**修复方法**:
- 默认 lookbackHours 从 24 降为 **4**
- self-heal 从 24 降为 **8**
- 最大值从 168 (7d) 降为 **48**

**边界与约束**:
```
规则 R1.4: lookbackHours 默认 = 4, self-heal = 8, 上限 = 48
           超出范围应 clamp，不 error
```

### 修复域 2：Reconcile 订单导入前置过滤 — Order 幂等

#### 2.1 `reconcileLiveAccountExchangeOrder` 增加交易所订单 skip 规则

**修复方法**:
在 `reconcileLiveAccountExchangeOrder` 函数中，`index.match` 返回 `!found` 时，增加以下 guard：

```
前置检查清单（按顺序，任一通过则 skip 该订单）：
1. 订单终态(FILLED/CANCELLED/REJECTED) 且 clientOrderId NOT 匹配本系统 order ID 前缀
   → skip, 原因: "non-system-order"
2. 订单终态 且 对应 symbol 当前 REST position = 0
   → skip, 原因: "closed-position-historical-order"
3. 订单终态(CANCELLED/REJECTED) 且 executedQty = 0
   → skip, 原因: "no-fill-cancelled"
```

**边界与约束**:
```
规则 R2.1: reconcile 创建 recovered order 前 MUST 通过以上全部 guard
           违反任一 → skip 并 emit structured log (WARN 级)

规则 R2.2: 本系统 order ID 前缀判断规则：
           strings.HasPrefix(clientOrderId, "order-") || 
           index.byClientOrderID 中存在
           不满足时视为非本系统订单

规则 R2.3: "对应 symbol 当前 REST position = 0" 判断基于
           syncLiveAccountFromBinance 拉取的 liveSyncSnapshot.positions，
           不是本地 positions 表（本地可能已经被污染）
```

#### 2.2 Order 创建添加 exchangeOrderId 存在性检查

**修复方法**:
在 `createRecoveredLiveOrderFromExchange` 之前，检查 `byExchangeOrderID` 索引。如果 index 中已存在该 exchangeOrderId 的 order，skip 创建，直接使用已有 order 进行 enrich/sync。

**边界与约束**:
```
规则 R2.4: 已存在的 order 走 enrich + sync，不走 create
           这消除了每次 reconcile 对同一 exchangeOrderId 创建新 orderID 的问题
```

### 修复域 3：Fill 幂等与 Position Apply 保护

> [!NOTE]
> 当前 fill 层已有 DB unique constraint + 内存 dedup + 数量限制。但这些保护在 order 层幂等失效（每次新 orderID）时会被绕过。修复域 2 解决 order 幂等后，fill 保护自然恢复有效。以下为额外加固。

#### 3.1 跨 Order 的 exchangeTradeId 全局去重

**修复方法**:
在 `filterExistingExecutionFills` 中，除了按 orderID 过滤，额外查询 **同 account** 下所有 fill 的 `exchangeTradeId`，如果已存在则跳过。

**边界与约束**:
```
规则 R3.1: 同一 account 下，同一 exchangeTradeId 的 fill 只允许创建一次
           不论 orderID 是否相同

规则 R3.2: 该规则仅对 exchange-sourced fills 生效
           (exchangeTradeId != "")
           synthetic/fallback fills 不受此约束
```

#### 3.2 `applyExecutionFill` 增加 reconcile 上下文感知

**修复方法**:
当 order 来源为 `reconcileRecovered: true` 时, `applyExecutionFill` 前检查：
- 当前 REST position 不为 0（该 symbol 确实有持仓）
- fill 的方向与当前持仓方向一致或已被上层确认为合理

**边界与约束**:
```
规则 R3.3: reconcile recovered order 的 fill 必须通过持仓一致性检查
           REST position = 0 时，禁止 apply 任何 fill
           这防止"已关闭持仓的历史订单"反向重开仓位
```

### 修复域 4：SyncLiveAccount 并发闸门与限速

#### 4.1 Per-Account Sync Mutex

**修复方法**:
在 `Platform` struct 中添加 per-account sync mutex：

```
type Platform struct {
    // ...
    liveAccountSyncMu   sync.Map  // key: accountID, value: *sync.Mutex
}
```

在 `SyncLiveAccount` 入口获取 mutex，如果已被持有则直接返回最新本地状态（非阻塞 tryLock 语义）。

**边界与约束**:
```
规则 R4.1: 同一 account 同一时刻只允许一个 SyncLiveAccount 执行
           并发触发时，后到者直接返回最新本地 account 状态
           不排队、不阻塞

规则 R4.2: ReconcileLiveAccount 同理，per-account 互斥
           且 ReconcileLiveAccount 执行期间不允许触发 SyncLiveAccount
           （或反过来：SyncLiveAccount 在 reconcile 进行中时跳过 adapter sync）
```

#### 4.2 Binance REST Rate Limiter

**修复方法**:
添加全局 Binance REST rate limiter，使用 token bucket：

```
限速参数:
- 默认: 30 requests/second (Binance IP limit 6000/min ≈ 100/s, 保守 30%)
- burst: 50
- 429 收到后: 退避至少 60 秒，期间拒绝新请求
```

**边界与约束**:
```
规则 R4.3: 所有 doBinanceSignedRequest / binanceSignedGET 调用前 
           必须从 rate limiter 获取 token

规则 R4.4: 收到 429 response 时:
           a. 立即标记 rate limiter 为 exhausted
           b. backoff 至少 60s
           c. 期间所有 Binance REST 请求直接返回 rate-limited error
           d. 不重试、不排队

规则 R4.5: 429 error 向上传播为 syncErr，触发 syncFailure 健康记录
           不允许静默忽略 429
```

### 修复域 5：恢复态状态边界显式化

> [!IMPORTANT]
> 此修复域对应 AGENTS.md §7.3 / §7.4 强制规则。

#### 5.1 Reconcile 导入的订单必须标记为 audit-only

**修复方法**:
对于确需导入的历史订单（通过了修复域 2 的全部 guard），在 metadata 中添加标记：

```json
{
  "reconcileRecovered": true,
  "reconcileAuditOnly": true,
  "reconcileSkipPositionApply": false  // 仅当 REST position > 0 时为 false
}
```

**边界与约束**:
```
规则 R5.1: reconcileAuditOnly = true 的订单:
           - 不参与 live session 的 lastDispatchedOrderId 推进
           - 不参与策略 planIndex 推进  
           - 不触发 auto-dispatch
           - 不触发 passive close
           - 用途仅为审计和 reckon

规则 R5.2: 未完成 REST 权威对账前 (livePositionReconcileGate.status != "synced"):
           - 不允许 auto-dispatch (已有 shouldBlockAutoDispatchForRecoveryIntent)
           - 不允许自动 passive close
           - 不允许推进策略
           - 只能进入 stale/conflict/close-only takeover
           (当前代码在 isLiveSessionBlockedByPositionReconcileGate 中已部分实现，
            但需确认覆盖了全部路径)
```

#### 5.2 Reconcile 完成后必须刷新 position gate

**修复方法**:
`ReconcileLiveAccount` 结束后已调用 `refreshLiveAccountPositionReconcileGate` — 确认无遗漏。但需增加：
- 如果 reconcile 创建了 recovered orders，gate 状态必须设为 `stale` 或 `requires-review`，**不允许自动设为 synced**
- 只有当 REST position 与本地 position 数量一致时，gate 才可以为 `synced`

**边界与约束**:
```
规则 R5.3: reconcile 后 position gate 状态判定:
           - REST position qty == local position qty → synced
           - 不一致 → stale (阻断策略推进, 等待 manual review)
           - reconcile 创建了新 recovered order → requires-review

规则 R5.4: gate 从 stale/requires-review → synced 的转换 
           只能通过下一次 SyncLiveAccount 的 
           refreshLiveAccountPositionReconcileGate 确认
           不允许 reconcile 内部自动清除
```

### 修复域 6：Submit 前二次校验

> [!NOTE]
> 对应 AGENTS.md §7.6 强制规则。当前已在 `ensureLiveExecutionBoundary` 和 `validateRecoveredPassiveCloseOrder` 中有部分实现。以下为补充。

#### 6.1 Reconcile recovered order 提交限制

**修复方法**:
reconcile recovered order 不应进入 submit 路径。这类 order 的 `source` = `live-account-reconcile`，`reconcileRecovered` = true。

**边界与约束**:
```
规则 R6.1: source = "live-account-reconcile" 的 order 禁止:
           - 进入 SubmitOrder 路径
           - 被 dispatchLiveSessionIntent 选为下单候选
           - 被 passive close watchdog 选为平仓候选

规则 R6.2: 提交订单前最终校验清单 (已有 + 新增):
           ☑ 当前 session status 允许执行
           ☑ order role 明确 (entry/exit/recovered-close)
           ☑ reduceOnly 正确
           ☑ hedge/one-way payload 正确
           ☑ positionSide 匹配
           ☑ 新增: order.source != "live-account-reconcile"
           ☑ 新增: reconcileGate.status == "synced"
```

## 四、回归测试要求

> [!IMPORTANT]
> 对应 AGENTS.md §7.8，必须覆盖以下场景。

### 必须覆盖的场景

1. **历史已平仓订单 skip 测试**: reconcile 拉回终态订单，对应 symbol REST position = 0，验证不创建 order、不创建 fill、不修改 position
2. **非本系统订单 skip 测试**: clientOrderId 不匹配 `order-*` 前缀，验证 skip
3. **同一 exchangeOrderId 重复 reconcile 测试**: 验证第二次 reconcile 时不创建新 order，复用已有 order
4. **跨 order exchangeTradeId 去重测试**: 同一 trade 被不同 orderID 引用，验证 fill 只创建一次
5. **并发 SyncLiveAccount 测试**: 两个 goroutine 同时 sync 同一 account，验证只有一个实际执行 REST 请求
6. **429 backoff 测试**: Binance 返回 429 后，验证后续请求被拒绝直至 backoff 期满
7. **reconcile gate 状态一致性测试**: REST position 与 local position 不一致时，验证 gate = stale
8. **recovered order 不进入 dispatch 测试**: source = "live-account-reconcile" 的 order 不被 dispatch
9. **symbol 收集范围测试**: 验证只收集有效 symbol（有持仓/open orders/running sessions），不从历史订单收集

## 六、一句话给编码 LLM 的关键约束

> **核心不变量**: `ReconcileLiveAccount` 的 `reconcileLiveAccountExchangeOrder` 只能对 **(a) 本系统提交的订单** 或 **(b) 对应 symbol 当前 REST 持仓 > 0 的非终态订单** 创建/更新本地 order 和 apply fill。所有终态 + 本地不存在 + REST position = 0 的交易所历史订单必须 skip，emit WARN log，不创建、不更新、不 apply。

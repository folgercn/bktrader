# 第二阶段：后端统一日志系统建设任务清单

本清单基于《20260415-日志查看台建设计划.md》整理，聚焦于后端为了支持完整的日志查看台（Phase 2）需要实施的各项改进与新接口开发。

> [!WARNING]
> **风险控制点：** 日志查询应当注重性能，**必须**支持基于 `cursor` 和 `limit` 的分页，绝对避免一次性加载全量日志导致服务 OOM。日志模块属于观测类，不应修改 `live` 核心交易与分发的底层逻辑！

---

## 1. 结构化业务事件日志 (Task Group B)
*核心目标：将分散于不同服务中的关键业务数据作为统一“事件”开放给前端日志台。*

- [ ] **梳理现有数据读取能力**：排查 `internal/store/postgres` 与 `memory` 模块，确认已有的 `StrategyDecisionEvent` (决策) 、`OrderExecutionEvent` (执行) 、`PositionAccountSnapshot` (快照) 的底层结构和查询情况。
- [ ] **开发事件聚合接口**：新增标准 REST API `GET /api/v1/logs/events`。
  - [ ] 实现针对三种业务事件的聚合查询逻辑。
- [ ] **实现高级查询与分页机制**：为 `/api/v1/logs/events` 加入完整的查询与过滤参数。
  - [ ] 支持根据实体筛选：`accountId`、`strategyId`、`runtimeSessionId`、`liveSessionId`、`orderId`、`decisionEventId` 等。
  - [ ] 支持基础过滤：`type`、`level`。
  - [ ] 支持分页与游标：`limit`、`cursor`、`from`、`to`（时间戳）。
- [ ] **开发前端对接层对接（联调）**：协助前端把接口数据转换并装载至 `LogStage.tsx` 中的统一事件列表。

---

## 2. 系统日志与请求日志 (Task Group C)
*核心目标：开放底层系统运行情况与 HTTP 请求的情况，填补当前纯靠 docker stdout 日志的盲区。*

- [ ] **明确 `slog` 日志抓取与存储方案**：因为当前后端 `cmd` 与 `internal/logging` 使用 `slog` 主要向标准输出写数据，需评估并引入日志留存机制。
  - *方案候选*：服务内建 Ring Buffer (环形缓冲区) 用于快速呈现、或是将文件滚动落盘 (`Rolling File`) 通过接口读取。
- [ ] **新增系统日志访问接口**：开发 `GET /api/v1/logs/system`。
  - [ ] 支持通过时间范围与 `level` 级别检索节点自身的系统与运行日志。
- [ ] **新增 HTTP 请求日志访问接口**：开发 `GET /api/v1/logs/http`。
  - [ ] 解析并暴露 `router.go` 中的 `requestLogMiddleware` 的输出数据。
  - [ ] 支持根据状态码 (`status`)、请求路径 (`path`)、耗时 (`duration_ms`) 等核心指标进行针对性的检索。

---

## 3. 实时跟踪事件总线 (Real-time Stream)
*核心目标：解决轮询（Polling）的延迟，实现控制台界面行云流水般的流畅推送。*

- [ ] **架构方案选择**：在 SSE (Server-Sent Events) 与 WebSocket 中二选一落实日志流动总线（目前建议**SSE优先**适配仅下发场景）。
- [ ] **新增实时流总线接口**：开发 `GET /api/v1/logs/stream` 推送通道。
  - [ ] 接入并整合多个模块的通道：包括 `alerts` 变化、`rt/live` 时间线 (timeline)、以及全新的 `execution events` 等实时变化广播。
  - [ ] 加入订阅退订与流速控制机制：确保下发不因前端卡顿造成后端积压。
  - [ ] （选做）添加流开关控制：根据前端实际需求允许暂挂和断开数据流，以防引起前端主监控图表性能抖动。
